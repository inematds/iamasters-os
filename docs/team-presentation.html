1. Contexto: porque se construiu

Após analisar as opções do mercado anglo-saxónico, detetámos um gap: existem sistemas operativos agênticos empacotados — repos Claude Code prontos para clonar e instalar, com skills curadas, brand context, multi-cliente e memória que aprende — mas nenhum em português.

iAmasters NÃO tinha equivalente próprio. Synapsis (o sistema de memória do Luis Pitik) cobre a camada de aprendizagem, mas não a arquitetura completa: faltam brand context partilhado, multi-cliente, skills curadas com colaboração, níveis de projeto formalizados, etc.

Decisão estratégica do Angel

Construir o iAmasters OS com três vantagens claras face ao resto do mercado:

  • Em português (o resto só inglês)
  • Com Synapsis embebido como engine de memória
  • Como repo público e neutro — sem info privada do Angel nem da academia. Qualquer operador IA pode cloná-lo, instalá-lo e usá-lo

Resultado: numa única sessão construíram-se as versões v0.1.0 → v0.4.0, com esqueleto + 12 skills + 4 templates verticais + scripts de gestão + docs operativos completos.

2. O que é o iAmasters OS

Um repositório Claude Code pronto para clonar e instalar que converte uma sessão vanilla do Claude Code num sistema operativo profissional para operadores de IA.

Para quem

Operador IA freelance

Serve vários clientes a partir do seu portátil. Quer padronizar entregáveis e profissionalizar sem criar estrutura corporate.

Empreendedor / PME

Automatiza o seu negócio com o Claude Code. Quer memória que persista entre sessões e skills que aprendam do seu contexto.

Agência pequena (2-10)

Quer padronizar a stack agêntica da equipa. Multi-cliente com voice e outputs separados por marca.

Formador / educador

Ensina Claude Code e precisa de um repo de referência que os alunos possam clonar e usar imediatamente.

O que dá no primeiro dia

  • Memória que persiste entre sessões — acabou o "explica-me a tua stack outra vez"
  • 12+ skills curadas instaladas, já com a tua marca dentro
  • Brand context (voz, posicionamento, ICP) auto-gerado em 30 minutos
  • Multi-cliente pronto para escalar (4 templates de vertical incluídos)
  • Sistema de aprendizagem contínua: o que repetes gradua-se a regra automática
  • Dashboard visual com métricas reais de uso do operador (herdado do Sinapsis)

4. Arquitetura: 3 camadas

O iAmasters OS combina três camadas distintas que trabalham em conjunto mas podem manter-se independentes:

Camada 1 · Sinapsis v4.5 (engine de memória)

vendor/sinapsis/ · vendorizado intacto · instalado em ~/.claude/

O engine que faz o Claude Code aprender de ti. Operator state global, instincts auto-aprendidos, skills on-demand, dashboard, dream cycle. Transplantado íntegro do repo do Luis Pitik sem modificar (preservamos os 112 tests upstream).

Camada 2 · OS (estrutura do repo)

brand-context/ · context/ · projects/ · clients/

A estrutura que adicionamos em cima do Sinapsis: brand context (voice, positioning, ICP), agent context (soul, user, learnings), projetos estruturados, multi-cliente com templates por vertical.

Camada 3 · Skills curadas + commands + scripts

.claude/skills/ · .claude/commands/ · scripts/

12 skills validadas em categorias marketing, operations, strategy, tools, visualization, _meta. 5 slash commands do OS. 4 scripts de gestão (install, add-client, update com conflict resolution, validate-skill).

Decisão chave: vendoring intacto. O Sinapsis mantém-se tal como o Luis publica. Nós não modificamos os paths nem os scripts dele. Quando o Luis publicar nova versão upstream, fazemos git pull em vendor/sinapsis/ e continuamos a funcionar. Zero risco de partir os 112 tests dele.

6. Como se instala

Prerequisitos

  • Claude Code CLI instalado e autenticado com plano Pro ou Max → claude.ai/code
  • Node.js 18+ → nodejs.org
  • Python 3.9+ (para hooks Sinapsis)
  • Git

Instalação rápida

git clone https://github.com/iamasters-academy/iamasters-os.git
cd iamasters-os
bash scripts/install.sh

O que o instalador faz (6 passos)

1

Verifica prerequisitos

Verifica que o Claude Code, Node.js, Python 3 e Git estão no PATH. Se falta algo crítico → exit com mensagem clara.

2

Deteta estado do Sinapsis

Se ~/.claude/skills/_catalog.json existe → Sinapsis já instalado, salta. Se não → continua para o passo 3.

3

Instala o Sinapsis (se for preciso)

Executa bash vendor/sinapsis/install.sh que é o installer original do Luis sem modificar. Configura os 7 hooks do Sinapsis em ~/.claude/settings.json (4 PreToolUse + 1 PostToolUse + 1 Stop + 1 PreCompact).

4

Inicializa a camada OS

Cria context/soul.md, context/user.md, context/learnings.md com templates. Cria .gitkeep em pastas vazias para o git as trackear.

5

Instala hook local do OS

Cria scripts/skill-change-detector.sh que deteta quando se adiciona/modifica uma skill e flagea para sync no próximo wrap-up.

6

Verifica instalação

Verifica que ~/.claude/skills/ e .claude/settings.json existem. Se tudo OK → mostra "próximo passo".

Tempo total: ~30 segundos.

7. Primeiro uso passo a passo (os primeiros 30 minutos)

1

Arrancar o Claude Code no repo

cd iamasters-os
claude

O Claude lê o CLAUDE.md raiz e executa a "MANDATORY first action": ler ~/.claude/skills/_operator-state.json. Como tem needsOnboarding: true → invoca automaticamente a skill meta-onboarding-wizard.

2

Onboarding wizard (3 minutos)

O wizard faz 7 perguntas:

  1. Avatar: single business / freelance / agência / formador
  2. Nível técnico: zero / intermédio / avançado
  3. Domínio principal: marketing / operations / strategy / dev / misto
  4. Stack atual (multi-select)
  5. Idioma de outputs cliente (se aplicável)
  6. Firecrawl API key (opcional)
  7. Identidade básica: nome, email, site/LinkedIn

Configura defaults inteligentes e preenche operator-state.json + context/user.md.

3

Brand voice setup (5-10 minutos)

O wizard pergunta se queres lançar marketing-brand-voice. Se disseres sim:

  • Colas os teus URLs públicos (site, LinkedIn, YouTube, blog)
  • Firecrawl scraping → extrai voice samples
  • 6 perguntas de calibração (tom dominante, vocabulário que evitas, frases-assinatura, jargão próprio, anti-modelo, modelo aspiracional)
  • Gera voice-profile.md + samples.md + 3 registos A/B/C

A tua marca fica "instalada" no OS. Qualquer skill marketing-* usá-la-á automaticamente.

4

Primeira tarefa real

Pedes algo concreto. Exemplo: "Escreve 3 versões de post LinkedIn sobre [o teu tema]".

O Claude:

  1. Invoca marketing-copywriting
  2. Lê o teu brand-context (voice + positioning + ICP)
  3. Gera 3 variações com ângulos distintos (storytelling / contraintuitivo / problema-solução)
  4. Passa CADA variação por tool-output-verifier (humanizer + brand voice + length + factuality)
  5. Entrega-te as 3 com scores e sugestões
5

Fecho com /wrap-up

/wrap-up

Gera daily summary, sincroniza skills se adicionaste alguma, propõe Git commit (espera o teu OK), diz-te o que retomar amanhã.

6

Próximos dias

Quando voltares a abrir o Claude Code no repo, executa /start-here. Recapitula-te o de ontem + propõe tarefa do dia. A memória persiste.

11. Os 4 templates verticais

Cada template inclui 6 ficheiros: README específico ao vertical + voice-profile + positioning + ICP (em brand-context/) + soul.md + user.template.md (em context/). Os placeholders {{CLIENT_NAME}} substituem-se automaticamente ao executar add-client.sh.

freelance-ia

Para: operador IA solo, serve 3-10 clientes, ticket 5-50K€/projeto.

Voice: profissional mas acessível (B com toques de C). Confiável, honesto, pragmático.

Positioning sugerido: vertical-specialist ou use-case-specialist. Anti-vendehumos.

ICP típico: PMEs 5-30 pessoas com dor concreta em horas perdidas / clientes em fila / colaborador chave que se vai.

Skills favoritas: marketing-copywriting, marketing-content-repurposing, tool-output-verifier.

agencia-marketing

Para: equipa 2-10, 5-30 clientes em simultâneo, MRR recorrente.

Voice: profissional criativo, equilíbrio formal-próximo (B com A para premium). Cada cliente final tem A SUA voice.

Positioning sugerido: agência especialista, não full-service. Foco vertical ou serviço.

ICP típico: PMEs/scaleups com MRR estabelecido a procurar crescer marketing.

Skills favoritas: marketing-content-repurposing (alta frequência output) + marketing-copywriting.

formador-online

12. Os 4 scripts

scripts/install.sh — installer principal do repo

Delega ao vendor/sinapsis/install.sh + inicializa a camada OS. 6 passos color-coded: prerequisitos → deteção Sinapsis → instalação se falta → init camada OS (soul.md, user.md, learnings.md) → hook local → verificação. Tempo total ~30 segundos.

scripts/add-client.sh — cria cliente desde template

Argumentos: nome-cliente (kebab-case obrigatório) + template (um dos 4 verticais ou "vazio"). Valida nome, copia template, substitui {{CLIENT_NAME}} em todos os .md, gera clients/<nome>/CLAUDE.md com overrides específicos.

bash scripts/add-client.sh acme-corp freelance-ia

Output: estrutura completa criada em clients/acme-corp/, pronta para configurar.

scripts/update.sh — pull com conflict resolution

Trata 4 cenários:

  1. Sem mudanças: já estás em dia, exit limpo
  2. Upstream atualizou, tu não modificaste: pull direto de ficheiros seguros
  3. Tu modificaste local, upstream não: keep local automaticamente
  4. Conflito na mesma linha: prompt caso-a-caso (k=keep / u=use upstream / d=diff / s=skip)

Backup automático em .backup/<timestamp>/ antes de tocar em nada. User data nunca se sobrescreve (brand-context, context, projects, clients, .env).

scripts/validate-skill.sh — valida skill antes de instalar

O motor do comando /install-skill. Passos:

  1. Faz parse do URL de GitHub (owner/repo + branch + subpath)
  2. Shallow clone para /tmp/iamasters-os-skill-validate-<hash>/
  3. Valida estrutura: SKILL.md presente, YAML frontmatter nas linhas 1-3
  4. Valida name kebab-case + description 30-500 chars + verbo de intenção
  5. Deteta scripts com padrões perigosos (rm -rf /, eval, dd if=, mkfs, curl|sh)
  6. Deteta credenciais hardcoded (regex API keys, tokens, passwords)
  7. Verifica conflito com skills locais do mesmo nome
  8. Output color-coded: OK · WARN · ERROR + recomendações de instalação
scripts/skill-change-detector.sh — hook local

Hook configurado em .claude/settings.json que dispara em PostToolUse de Edit/Write. Se a edição foi em .claude/skills/, escreve flag em .claude/.skills-pending.json para que o próximo /wrap-up sincronize o catálogo e atualize o registry do CLAUDE.md.

14. Fluxo multi-cliente típico

Criar cliente novo

bash scripts/add-client.sh acme-corp freelance-ia

Isto cria clients/acme-corp/ com estrutura completa: brand-context, context, projects, briefs, CLAUDE.md de overrides.

Trabalhar dentro de um cliente

O Claude Code arranca no contexto do cliente:

  • Lê o CLAUDE.md raiz do repo (instruções gerais)
  • + clients/acme-corp/CLAUDE.md (overrides específicos)
  • + clients/acme-corp/brand-context/ (voice/positioning/ICP próprios do cliente)
  • + clients/acme-corp/context/ (soul/user do cliente)

Exemplo de dia com 3 clientes

# Cliente A (freelance-ia)
cd clients/acme-corp && claude
> "Cria post LinkedIn sobre case study X"
   → marketing-copywriting gera com voice de Acme
> /wrap-up
exit

# Cliente B (agencia-marketing)
cd ../widget-shop && claude
> "Repurpose último vídeo do CEO em 5 peças"
   → marketing-content-repurposing
> /wrap-up
exit

# Cliente C (consultoria-b2b)
cd ../north-star-consult && claude
> "Redige proposta comercial para novo lead"
   → marketing-copywriting registo A formal
> /wrap-up
exit

Best practices

  1. Sempre cd antes de claude. Não trabalhar no cliente X desde a raiz.
  2. Wrap-up ao mudar de contexto. Daily summary separa-se por cliente.
  3. Brand voice por cliente, não partilhar entre clientes.
  4. Uma skill custom = um cliente, ou promovê-la para a raiz se aplicar a vários.
  5. Backup do cliente regularmente se tiver info crítica (não do git público).

15. Decisões técnicas chave

Vendoring intacto do Sinapsis

Decidimos NÃO modificar os paths hardcoded do Sinapsis ($HOME/.claude/skills/...) em vendor/sinapsis/. Razões:

  • O Sinapsis tem 112 tests a passar com esses paths. Modificá-los parte upstream.
  • Quando o Luis publicar nova versão, fazemos git pull em vendor e continuamos a funcionar.
  • O installer do Sinapsis está pensado para instalação global. Forçá-lo a in-repo era muito trabalho de reescrita sem valor real.

Implicação: o Sinapsis vive em ~/.claude/ (global do operador). O repo iAmasters OS só adiciona camadas em cima. Cross-project memory do Sinapsis aplica-se a todos os repos do operador, não só ao iAmasters OS.

Repo público e neutro (sem info privada)

Decisão explícita do Angel: o repo é público (privado em v0 mas pensado para abrir) e neutro. Qualquer operador pode cloná-lo sem que se infiltre info do Angel nem da iAmasters academy.

Implicação: os templates de cliente estão em branco com placeholders {{CLIENT_NAME}}. Os exemplos nos docs são genéricos. As skills não assumem nenhum avatar concreto.

Sinapsis embedded como engine, não como camada secundária

Os concorrentes do mercado anglo-saxónico têm um learnings.md plano por skill. Isso é um primeiro passo muito básico. O Synapsis já está vários níveis acima: instincts com confidence score, evolução para skills/regras, projects registry estruturado, dream cycle de hygiene, dashboard.

Decisão: transplantamos o Sinapsis íntegro como engine principal, não como camada secundária. Essa é a nossa maior vantagem competitiva.

Padrão canónico de skill definido e enforced

O meta-skill-creator define as regras não-negociáveis para skills novas:

  • YAML frontmatter completo (name kebab-case + description 50-500 chars com verbo de intenção)
  • Progressive disclosure (knowledge extenso em references/)
  • Steps numerados e testáveis
  • Skill collaboration explícita
  • Output verifier gate se gera entregáveis
  • Learnings hook ao fecho

Implicação: qualquer skill nova (interna ou instalada via /install-skill) deve cumprir o padrão. scripts/validate-skill.sh faz check automático.

3 registos de voice (A/B/C) em vez de 1 voice profile genérico

O standard habitual é 1 voice profile genérico. Mas a realidade operativa é que escrevemos muito diferente em email premium vs LinkedIn vs WhatsApp. Se o OS não o modela, todos os outputs saem homogéneos.

  • A formal: email cliente premium, proposta comercial, contrato
  • B divulgativo: post LinkedIn, blog, video script, newsletter (95% do conteúdo público)
  • C próximo: WhatsApp comunidade, DM, comentários redes

O marketing-copywriting auto-deteta o registo correto conforme plataforma + purpose.

tool-output-verifier como gate obrigatório

Todas as skills marketing-* invocam tool-output-verifier no último passo. Não é opcional.

Porquê: em operação real, os operadores saltam os gates "porque tenho pressa" e entregam AI slop ao cliente. Torná-lo obrigatório força o padrão profissional.

Threshold por plataforma: humanizer ≥8 para email premium · ≥7 para LinkedIn/blog · ≥6 para WhatsApp comunidade. Brand voice match sempre ≥7.

Backups automáticos em update.sh

Antes de aplicar qualquer update, scripts/update.sh cria backup automático em .backup/<timestamp>/ com: .claude/skills/, brand-context/, context/, projects/, clients/, .env, .claude/settings.json.

Política: nunca confiar que algo "não se vai tocar". Backup sempre, recovery 1-comando.

User data nunca no git público

O .gitignore estendido exclui:

  • brand-context/voice/voice-profile.md (e samples, registers)
  • brand-context/positioning/positioning.md
  • brand-context/icp/icp.md
  • brand-context/assets/
  • context/user.md e context/daily-memory/
  • projects/briefs/* e outputs por skill
  • clients/* exceto _templates/
  • .env

Só se commitam: README, soul.md, learnings.md (genérico), templates vazios.

Validate-skill não elimina TMP automaticamente

Quando alguém tenta instalar uma skill externa com /install-skill, o script valida mas NÃO elimina o TMP dir. Razões:

  • O operador pode inspecionar manualmente a skill descarregada
  • Se a validação falhar, podia haver falso positivo e querer rever
  • Deixa o operador decidir quando limpar (rm -rf /tmp/iamasters-os-skill-validate-*)
Curated lists são opinated, não inclusivas

As listas docs/mcps-curated.md e docs/skills-recommended.md NÃO são catálogos exaustivos. Só incluem o que validámos em produção ≥2 semanas. Razões:

  • Inclusão sem filtro causa decision paralysis
  • O valor real de uma curated list é o filtro
  • "Se não sei, não recomendo" é melhor que "metemos por precaução"

Para contribuir para as listas é preciso experiência real documentada (ver secção "Como contribuir" em cada doc).
